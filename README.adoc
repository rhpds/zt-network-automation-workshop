= Zero Trust Network Automation Workshop
:toc: macro
:toc-title: Table of Contents
:icons: font

image:https://img.shields.io/badge/License-MIT-yellow.svg[License: MIT,link=LICENSE]

A hands-on workshop for learning **network automation with Ansible Automation Platform**. Participants configure multi-vendor routers (Cisco IOS-XE, Arista EOS, Juniper Junos) running on link:https://containerlab.dev/[Containerlab^], then graduate to managing those devices through **AAP job templates, surveys, RBAC, and workflows**.

Designed for link:https://demo.redhat.com[Red Hat Demo Platform (RHDP)^] but adaptable to any environment with two Linux VMs and a container runtime.

toc::[]

== Quick start

[source,shell]
----
# On the control VM, after AAP is up:
cd lab-automation
export CONTROLLER_PASSWORD='...'
ansible-playbook -i inventory/lab.yml playbooks/site.yml
----

This deploys the Containerlab topology on the **containerlab** VM and bootstraps Automation Controller with the workshop's SCM project and job templates.

For manual Containerlab CLI usage (no Ansible), see link:README-containerlab.md[README-containerlab.md].
For detailed lab-automation docs, see link:lab-automation/README.md[lab-automation/README.md].

== Features

* **Multi-vendor network lab** -- Cisco IOS-XE, Arista EOS, and Juniper Junos virtual routers on Containerlab
* **Ansible Automation Platform integration** -- pre-provisioned Controller with SCM project, job templates, surveys, RBAC roles, and workflows
* **Auto-resume on boot** -- a systemd service (`containerlab-resume`) automatically re-deploys the last active topology when the VM restarts (see <<auto-resume>>)
* **Network report dashboard** -- Podman-hosted web report accessible via a dedicated browser tab
* **VS Code in-browser** -- code-server with syntax highlighting and integrated terminal
* **RHDP-native** -- one-click provisioning on Red Hat Demo Platform with Showroom UI tabs

== Lab automation

Playbooks under `lab-automation/` handle:

1. **Deploying** the multi-vendor router topology on the containerlab VM
2. **Bootstrapping** AAP Controller -- waiting for the API, creating the Default organization, syncing the SCM project, and creating job templates with RBAC
3. **Optionally** gathering facts from the virtual routers

[source,shell]
----
cd lab-automation
export CONTROLLER_PASSWORD='...'
ansible-playbook -i inventory/lab.yml playbooks/site.yml
----

`setup-automation/setup-control.sh` runs this automatically at first boot when it finds `lab-automation/` under the workshop repo.

Logs: `/tmp/lab-automation-site.log`, `/tmp/lab-automation-collections.log`.

[[auto-resume]]
== Containerlab auto-resume on boot

Virtual routers do not survive a VM reboot because Containerlab topologies run as containers. A **systemd service** automatically re-deploys the last active topology when the containerlab VM starts or resumes.

=== How it works

1. **State file** -- When a topology is deployed (via the Ansible playbook or at first boot), the topology directory path is written to `/etc/containerlab/last-topology`.
2. **systemd service** -- `containerlab-resume.service` runs at boot (after `network-online.target`). It reads the state file and runs `containerlab deploy --reconfigure` in the recorded directory.
3. **Startup time** -- After the service completes, routers report `health: starting` and need 1-3 minutes to fully boot before SSH is available. This is normal.

=== Verifying the service

[source,shell]
----
# Check service status
sudo systemctl status containerlab-resume.service

# View service logs
journalctl -u containerlab-resume.service

# See which topology is recorded
cat /etc/containerlab/last-topology

# Check router health
sudo containerlab inspect
----

=== Implementation details

* **Helper script**: `/usr/local/bin/containerlab-resume` -- reads the state file and runs the deploy
* **Unit file**: `/etc/systemd/system/containerlab-resume.service` -- `Type=oneshot`, `RemainAfterExit=true`
* **State file**: `/etc/containerlab/last-topology` -- single line containing the absolute path to the topology directory
* **Installed by**: `setup-automation/setup-containerlab.sh` (`install_clab_resume_service` function)
* **State written by**: `lab-automation/playbooks/1_multi_vendor_router_up.yml` (after successful `containerlab deploy`)

== Setup automation

First-boot scripts in `setup-automation/` run once when RHDP provisions the VMs:

[cols="1,3",options="header"]
|===
| Script | Purpose

| `setup-control.sh`
| Installs collections, runs `lab-automation/playbooks/site.yml`, bootstraps AAP

| `setup-containerlab.sh`
| Clones the workshop repo, pushes SSH keys to control, configures router SSH access, installs the `containerlab-resume` systemd service

| `setup-vscode.sh`
| Registers with RHSM, installs Podman, configures code-server
|===

Logs are written to `/tmp/progress.log` on each VM.

== Runtime automation

Per-module scripts under `runtime-automation/` power the Showroom **Setup**, **Solve**, and **Validation** buttons:

[source]
----
runtime-automation/
├── module-01/
│   ├── setup-control.sh
│   ├── solve-control.sh
│   └── validation-control.sh
├── ansible.cfg
└── main.yml
----

Script names must match the `virtualmachines[].name` in `config/instances.yaml` (e.g. `setup-control.sh` runs on the **control** VM).
